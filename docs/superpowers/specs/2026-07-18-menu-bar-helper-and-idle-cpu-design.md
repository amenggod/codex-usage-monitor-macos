# 独立菜单栏助手与空闲 CPU 优化设计

## 背景

Codex Usage Monitor 当前存在两个相互独立、但会共同影响常驻体验的问题。

第一，主程序已经创建了 `NSStatusItem`，设置中的菜单栏开关也处于开启状态，但图标仍不在当前屏幕和活动空间内。最小 AppKit 与 SwiftUI 探针证明：使用其他 Bundle ID 时状态项正常可见，使用 `com.amenggod.CodexUsageMonitor` 时则被 macOS 放到屏幕外。切换 `autosaveName`、重建状态项、重启 `SystemUIServer` 或改用 `MenuBarExtra` 都不能恢复。该状态与主程序身份绑定，并曾受到 Only Switch 菜单栏折叠功能影响。

第二，空闲状态下主程序约占用 22.5% CPU，其启动的 `codex app-server` 子进程约占用 13%。代码中存在两个持续工作源：主窗口与菜单栏弹窗通过 `TimelineView` 每秒重绘，主程序也每秒刷新一次状态项；实时额度服务则让 `codex app-server --stdio` 长期驻留。两者都不需要为了分钟级额度同步而持续运行。

## 目标

- 恢复稳定可见的 macOS 菜单栏入口。
- 点击菜单栏图标后显示轻量额度面板，并可打开完整主程序。
- 菜单栏、主程序和桌面小组件使用同一份共享快照。
- 主程序继续在后台增量统计 Token，并每分钟同步实时额度。
- 手动刷新立即触发一次额度同步，并把新快照传播给所有界面。
- 空闲稳定后，主程序与菜单栏助手合计 CPU 通常低于 2%。
- 两次额度同步之间不常驻额外的 `codex app-server` 进程。
- 额度请求失败时保留最近可信数据，同时明确标注陈旧或不可用状态。

## 非目标

- 不绕过或修改 Only Switch 的内部设置。
- 不保证第三方菜单栏管理工具主动再次隐藏新助手时仍能强制显示。
- 不用浮动桌面窗口替代原生 WidgetKit 小组件。
- 不在本次修改中重写会话文件索引或 SQLite 聚合架构。
- 不追求秒级额度轮询；Codex 上游额度本身按分钟级同步。
- 不把凭据、提示词、完整会话内容或完整 app-server 响应写入 App Group。

## 方案选择

采用“独立菜单栏助手 + 事件驱动界面 + 按需额度进程”。

独立助手使用新的 Bundle ID，避开绑定在主程序身份上的隐藏状态。主程序仍是唯一的数据生产者；助手只读取脱敏后的 App Group 快照。主程序不再创建旧的状态项。

界面不再每秒重绘。实时额度仍每分钟同步，但 `codex app-server` 仅在单次请求期间存在，请求结束后立即退出。相比单纯降低界面刷新频率，这一方案同时消除主程序和子进程的持续空闲占用，又不会牺牲现有分钟级准确性。

## 组件设计

### 主程序

主程序继续负责：

- 监听 Codex 会话文件并增量写入 SQLite。
- 聚合今日、七天、全部和项目 Token 统计。
- 每分钟读取实时额度，执行阈值提醒。
- 把脱敏后的最新数据写入 App Group。
- 根据“显示顶部菜单栏”设置启动或停止菜单栏助手。
- 处理来自助手的打开面板、设置和手动刷新请求。

主程序删除 `AppKitMenuBarController` 的运行时职责，不再以 `com.amenggod.CodexUsageMonitor` 创建状态项。

### 菜单栏助手

新增嵌入主安装包的 LSUIElement 应用目标：

- Bundle ID：`com.amenggod.CodexUsageMonitor.MenuBar`
- 安装位置：主应用的 `Contents/Library/LoginItems`
- 权限：仅使用与主程序和 Widget 相同的 App Group
- 数据依赖：`CodexUsageShared` 中的脱敏快照类型和存储

助手创建一个紧凑模板图标。点击后弹出轻量面板，显示：

- 今日 Token
- 总 Token
- 周剩余额度
- 可用时的 5 小时额度
- 前三项目用量
- 数据更新时间和可信状态

面板提供“刷新”“设置”“打开完整统计”和“退出程序”。刷新与设置通过 `codexusagemonitor://` URL 唤起主程序；退出程序会结束主程序与助手。助手不读取 Codex 会话文件、不打开 SQLite，也不启动 app-server。

### 助手生命周期

主程序在完成启动后读取 `MenuBarVisibilityStore`：

- 开关开启：通过 `NSWorkspace` 启动嵌入的助手。
- 开关关闭：向助手发送退出通知；助手移除状态项并退出。
- 登录后台启动：按同一偏好恢复助手，不打开主窗口。
- 设置运行时改变：立即执行对应启停操作。
- 主程序意外退出：助手保留最近快照和重新打开主程序的入口。
- 用户从助手选择退出：两端一起退出。

启动管理器必须幂等；重复启动通知不能产生多个助手实例或多个状态项。

## 数据流与跨进程同步

`WidgetUsageSnapshot` 继续作为主程序、Widget 和菜单栏助手的共享数据契约。现有字段已经包含今日 Token、总 Token、前三项目、实时额度、新鲜度和数据状态，不需要复制数据库或主程序的 ViewModel。

同步流程：

1. 会话文件发生有效变化，或实时额度请求完成。
2. 主程序聚合数据并原子写入 App Group 快照。
3. 写入成功后发布无载荷的 Darwin 通知。
4. 菜单栏助手收到通知后重新读取快照并更新可见内容。
5. WidgetKit 继续按现有“可见指纹变化”策略申请重载。
6. 助手每 60 秒低成本读取一次快照，作为通知丢失和更新时间文案的兜底。

Darwin 通知只作为唤醒信号，不携带 Token 或额度内容。真实数据始终来自原子写入的 App Group 文件，避免通知与文件内容竞态。

## CPU 优化

### 事件驱动界面

删除 `UsagePopoverView` 的每秒 `TimelineView`。主窗口只在 `UsageViewModel` 快照变化时重绘；额度服务每分钟产生的新状态足以更新重置时间和新鲜度文案。

删除主程序状态项每秒刷新计时器。新助手只在共享快照通知到达或 60 秒兜底时更新图标、辅助功能标签和面板模型。面板关闭时不运行单独的动画或秒级时间线。

### 按需 app-server

`CodexRateLimitService` 保留每分钟轮询和立即手动刷新，但改变传输生命周期：

1. 启动 `codex app-server --stdio`。
2. 发送 `initialize`。
3. 发送 `account/rateLimits/read`。
4. 解码并保存实时额度。
5. 无论成功、失败或超时，都关闭输入管道并终止本次进程。

由于进程不再常驻，服务不依赖 app-server 推送通知；一分钟轮询和手动刷新是唯一刷新入口。

自动轮询、重试和手动刷新采用单航班机制。同一时刻至多存在一个请求任务；后续调用等待已有任务完成，不能并发启动多个 `codex` 进程。失败后的退避策略继续使用 5 秒、30 秒、120 秒，但每次尝试仍是独立、可清理的短生命周期进程。

## URL 与操作路由

主程序扩展现有 URL 路由：

- `codexusagemonitor://dashboard`：打开完整统计。
- `codexusagemonitor://refresh`：后台启动主程序并立即执行一次 Token 与实时额度刷新；不强制打开窗口。
- `codexusagemonitor://settings`：激活主程序并打开设置。

助手在发送操作后继续显示最近快照。刷新完成后，App Group 写入和 Darwin 通知会自动更新面板与 Widget。

## 错误处理

- 助手包缺失或无法启动：设置页显示明确错误，不影响主程序和 Widget。
- 助手首次读不到快照：显示“等待 Codex Usage Monitor 同步”，所有打开主程序的入口仍可用。
- 快照损坏：忽略损坏内容，保留进程内最后一次有效快照并显示陈旧状态。
- Darwin 通知丢失：60 秒兜底读取恢复一致性。
- app-server 启动、初始化、请求或解码失败：在 `defer` 清理进程，保存安全错误文案，保留最近可信额度。
- 请求超时：结束进程并进入现有退避策略，不能留下孤儿子进程。
- 主程序与助手同时收到退出请求：退出操作幂等，不重复发送或阻塞终止。

## 测试策略

实施遵循测试驱动开发，每个行为先通过失败测试证明缺失，再写最小实现。

### 菜单栏助手

- 偏好开启时只启动一个助手实例。
- 偏好关闭时发送一次退出并停止助手。
- 登录后台启动按偏好恢复助手且不打开主窗口。
- 快照通知触发重新读取；相同快照不产生多余 UI 更新。
- 60 秒兜底能恢复丢失通知后的数据。
- 损坏或缺失快照显示安全占位内容。
- 刷新、设置、完整统计和退出操作生成正确路由。
- Helper target 使用独立 Bundle ID、LSUIElement 和正确 App Group entitlement。

### 实时额度与 CPU 行为

- 一次成功刷新严格执行 start、initialize、read、stop。
- 初始化失败、读取失败、超时和取消都会 stop。
- 自动与手动刷新重叠时只启动一个传输进程。
- 一分钟轮询继续发布新额度；手动刷新立即发布。
- 失败时保留最近可信额度并更新新鲜度状态。
- 主窗口和菜单栏助手不存在一秒周期刷新策略。
- 主程序退出或服务停止后没有继续运行的轮询任务。

### 回归验证

- 全部 Swift 测试通过。
- Release Xcode 构建、嵌入关系、签名与 App Group entitlement 通过。
- 安装包中同时存在主程序、Widget、登录项和菜单栏助手。
- 点击 Widget 和菜单栏助手都能打开同一主程序。
- 手动刷新后主程序、共享快照、助手和 Widget 显示同一额度。

## 性能与现场验收

安装新版本后进行以下现场检查：

1. 菜单栏助手的状态项窗口位于当前屏幕，且 `occlusionState` 表示可见。
2. 关闭主窗口和弹窗，等待两分钟进入稳定空闲状态。
3. 活动监视器中主程序与助手合计 CPU 通常低于 2%。
4. 两次同步之间不存在主程序启动的常驻 `codex app-server`。
5. 到达一分钟同步点或手动刷新时允许短暂 CPU 峰值；请求结束后恢复空闲基线。
6. 完成一次 Codex 会话写入后，Token 统计更新且没有持续高占用。

CPU 百分比受机器负载影响，因此以连续观察到的稳定区间为准，而不是单个瞬时采样。
