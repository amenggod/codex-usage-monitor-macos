# Codex Usage Monitor 数据修复与桌面卡片设计

日期：2026-07-15  
状态：已确认，等待实施计划  
目标平台：macOS 14 及以上

## 1. 背景与问题

当前版本假设一个 Codex JSONL 文件只对应一个会话，并在 `sessions.file_key` 上建立唯一约束。新版 Codex 在分支或继承会话中会把父会话历史复制进新的 JSONL 文件，因此：

- 一个文件可能依次出现父会话和子会话的 `session_meta`；
- 同一段父会话 Token 历史可能出现在多个文件；
- 第二个会话写入同一 `file_key` 时会触发 SQLite 唯一约束错误；
- 扫描批次遇到该错误后停止，其他文件无法进入索引；
- 即使移除约束，继续以文件路径和字节偏移作为事件标识也会重复统计分支复制的历史。

当前菜单栏标签同时显示多个数值，在菜单栏拥挤或带刘海的 Mac 上容易被隐藏。新版本需要提供不占菜单栏的桌面卡片，同时允许用户保留菜单栏入口。

## 2. 目标

- 实时、准确地显示 Token 消耗量、有效限额剩余量和项目排行。
- 正确处理单文件多会话与跨文件复制的分支历史。
- 新增桌面常驻卡片，不占 Dock，普通应用窗口可以覆盖它。
- 设置中支持仅桌面卡片、仅菜单栏、两者同时显示。
- 只对当前存在且有效的限额窗口发送 20% 和 10% 提醒。
- 不读取、保存或输出提示词、回复、工具输出或凭据。

## 3. 非目标

- 不实现 WidgetKit 系统小组件。
- 不实现始终置顶的悬浮层。
- 不增加云端同步、遥测、OpenAI API Key 或远程统计。
- 不增加 Token 成本估算、团队看板或多设备聚合。
- 本次工作不包含 Apple Developer ID 签名与公证，但构建结果继续保持可签名结构。

## 4. 数据模型 v2

### 4.1 会话与文件解耦

`sessions` 以 Codex 会话 ID 为唯一主键，不再把 `file_key` 设为唯一归属。会话保存开始时间、规范化项目键、项目显示名和可选完整路径。

`file_cursors` 继续以文件标识为主键，并保存：

- 当前路径；
- 已处理字节偏移；
- 最后修改时间；
- 已处理位置对应的当前会话 ID。

扫描从文件中间继续时，使用游标中的当前会话 ID。遇到新的 `session_meta` 后立即切换当前会话，并在成功提交文件批次时更新游标。

### 4.2 逻辑 Token 事件

Token 事件不再使用文件路径和字节偏移作为身份。逻辑事件键由以下规范化字段生成 SHA-256：

- 会话 ID；
- 事件时间；
- `last_token_usage` 的五项 Token 值；
- `total_token_usage` 的五项 Token 值。

同一父会话历史被复制到多个分支文件时会生成相同事件键，只保留一条。文件路径、提示词正文和未知字段不参与事件键。

`usage_events` 保存解析后的单次用量和累计用量字段。若存在有效的 `last_token_usage`，其 `total_tokens` 是该事件的权威增量。若只有累计快照，则按同一会话的事件时间顺序计算非负差值。插入更早的累计事件或恢复历史文件后，重新计算受影响会话的累计差值，避免扫描顺序造成少算或多算。

重复事件在更新累计基线和限额快照之前被识别；重复事件不会移动累计基线。

### 4.3 限额快照

限额仍是账户级数据，不按项目拆分。限额快照按观察时间只接受更新的数据。

- `window_minutes == 300` 识别为 5 小时窗口；
- `window_minutes == 10080` 识别为周窗口；
- 其他窗口保留服务端标签，不冒充已知窗口。

快照的 `resets_at` 已经过期且没有更晚观察值时，不再视为实时数据。

## 5. 扫描与恢复

### 5.1 文件级原子处理

每个文件独立处理和提交。文件成功处理后才保存新游标；发生文件读取、数据库或结构错误时，该文件游标保持不变，其他文件继续扫描。

单行损坏或未知事件继续采用跳过策略。文件级失败记录脱敏摘要、失败次数和最近失败时间，不记录正文或完整用户路径。

### 5.2 实时更新

目录监听继续覆盖 `sessions` 和 `archived_sessions`。文件变化经短防抖后只读取新增完整行，目标是从日志落盘到界面更新不超过 2 秒。

失败文件每 30 秒自动重试；用户也可以手动重试。文件截断或替换时清空该文件的偏移和当前会话 ID，从头扫描并依靠逻辑事件键去重。

### 5.3 数据可信状态

所有发现文件均处理成功时，界面显示实时状态。存在失败文件时继续显示已成功处理的数据，但明确标记“部分数据等待恢复”，不把部分统计标记为完全实时。

数据库升级或重建期间显示进度并保留最后一次可用界面快照。首次升级没有可信快照时显示“正在重建本地统计”，不显示临时的零值作为最终结果。

## 6. v1 到 v2 迁移

Token 索引完全可以从本地 Codex 日志重建，因此升级采用确定性迁移：

1. 读取并保留通知回执；
2. 保留 UserDefaults 中的通知选择、显示模式、窗口位置和登录启动设置；
3. 删除 v1 会话、Token、累计基线、限额和文件游标索引；
4. 建立 v2 表结构；
5. 恢复通知回执；
6. 扫描全部活跃和归档日志；
7. 全部必要文件处理完成后发布新的可信快照。

迁移失败时保留错误状态和可重试入口，不删除原始 Codex 日志。

## 7. 限额显示与通知

### 7.1 5 小时窗口

只有存在尚未过期的 300 分钟限额快照时才显示 5 小时卡片。如果 Codex 不再返回该窗口，或者旧快照的重置时间已经过去，则：

- 桌面卡片和大面板完全隐藏 5 小时内容；
- 菜单栏不显示 5 小时文字；
- 不发送 5 小时 20% 或 10% 提醒。

以后再次读取到有效 300 分钟窗口时，显示和提醒自动恢复。

### 7.2 周窗口

有效周窗口显示剩余比例和重置时间。周快照过期后显示“等待新的周限额数据”，旧数值不作为实时剩余量展示，也不触发提醒。

### 7.3 通知

通知仅评估当前存在且有效的窗口。每个限额 ID、重置时间和阈值组合只发送一次。剩余量首次低于 20% 时提醒，首次低于 10% 时再次提醒；新重置窗口可以重新提醒。用户设置的总开关和两级阈值开关继续生效。

## 8. 显示模式

新增 `DisplayMode`：

- `desktop`：仅桌面卡片；
- `menuBar`：仅菜单栏；
- `both`：桌面卡片与菜单栏同时显示。

新安装和本次升级没有已保存选择时默认使用 `desktop`。设置通过 UserDefaults 持久化，切换立即生效。

菜单栏使用可动态插入和移除的 `MenuBarExtra`。没有 5 小时窗口时只显示周剩余量；两个已知窗口都不可用时显示简短的 `Codex --`。

## 9. 桌面卡片

### 9.1 窗口行为

桌面卡片由专用 `NSPanel` 控制器承载 SwiftUI 内容：

- `LSUIElement` 保持启用，不显示 Dock 图标；
- 使用普通窗口层级，不始终置顶，其他应用可以覆盖；
- 无系统标题栏，采用圆角半透明卡片和系统阴影；
- 可从空白区域拖动；
- 保存窗口位置，并在屏幕或分辨率变化后把窗口限制回可见区域；
- 不提供普通关闭按钮，避免仅桌面模式下失去入口；
- 设置页和退出按钮始终可达；
- 再次从 Finder 启动应用时，把桌面卡片显示到当前桌面并前置一次。

### 9.2 紧凑与展开

紧凑状态约为 `340 × 220`，显示：

- 当前有效限额；
- 今日总 Token；
- 最近更新时间或数据状态；
- 展开、设置和退出入口。

5 小时窗口隐藏后卡片自动收缩，不保留空白占位。

展开状态约为 `520 × 480`，复用现有大面板能力：

- 今日、近 7 天和全部筛选；
- 当前范围总 Token；
- 项目排行；
- 数据状态、重试、重建和设置。

紧凑或展开状态持久化。展开不是任意尺寸缩放，保证信息布局稳定。

## 10. 设置页

设置页新增“显示位置”分段选择：

- 仅桌面卡片；
- 仅顶部菜单栏；
- 桌面卡片与菜单栏。

原有设置继续保留：

- 登录时启动；
- 通知总开关；
- 20% 提醒；
- 10% 提醒；
- 重建本地索引。

设置页显示最近成功更新时间、失败文件数量和重建进度，但不显示未经脱敏的完整日志路径。

## 11. 组件边界

- `UsageRepository`：v2 模式、逻辑事件去重、累计差值重算和查询。
- `SessionScanner`：单文件解析、当前会话切换、游标恢复和文件级结果。
- `IngestionCoordinator`：发现文件、隔离失败、重试、重建进度和更新发布。
- `LimitAvailabilityPolicy`：判断已知限额是否存在、有效或过期。
- `DisplayModeStore`：持久化三种显示模式。
- `DesktopCardWindowController`：创建、显示、定位和恢复桌面卡片。
- `DesktopCardView`：紧凑内容和展开入口。
- `UsagePopoverView`：展开后的完整统计面板，由桌面与菜单栏复用。
- `UsageViewModel`：统一向两种界面发布可信快照、部分失败和重建状态。

## 12. 测试设计

所有测试使用合成日志和临时目录，不提交真实用户会话。

### 12.1 数据测试

- 一个文件先出现父会话、再出现子会话，Token 分别归属正确项目；
- 多个分支文件复制相同父会话历史时只统计一次；
- 同一逻辑事件位于不同路径和偏移时事件键一致；
- 重复事件不会改变累计基线；
- 只有累计快照时按事件时间计算差值；
- 较早累计事件后到达时重新计算结果且不产生负值；
- 增量读取从游标恢复正确当前会话；
- 文件截断和替换后重扫不重复；
- 一个文件失败不阻断其他文件，恢复后补齐统计；
- v1 迁移到 v2 后通知回执得到保留。

### 12.2 限额与通知测试

- 有效 5 小时和周窗口正常显示；
- 缺失或过期的 5 小时窗口完全隐藏；
- 过期周窗口不显示旧剩余值；
- 缺失窗口不发送通知；
- 20% 和 10% 阈值、通知去重和重置后重新提醒。

### 12.3 界面与窗口测试

- 三种显示模式立即生效并持久化；
- 默认模式为仅桌面卡片；
- 桌面模式不插入菜单栏项目；
- 菜单栏模式不创建桌面卡片；
- 两者模式同时显示且共享同一视图状态；
- 桌面卡片保存并恢复位置，越界位置得到修正；
- 紧凑和展开状态正确恢复；
- 再次启动时重新显示桌面卡片；
- 隐藏 5 小时卡片后布局没有空白占位；
- 读取失败时显示可理解的文字摘要。

## 13. 验收标准

- 现有真实日志可以完成重建，不再出现 `sessions.file_key` 唯一约束错误。
- 分支复制历史不会增加总 Token；新增 Token 在 2 秒内显示。
- 今日、7 天、全部及项目排行与去重后的日志统计一致。
- 5 小时窗口缺失或过期时不显示、不占位、不通知。
- 有效周限额与通知按设置工作。
- 默认启动显示桌面卡片且不占菜单栏；设置可切换三种模式。
- 桌面卡片可拖动、记住位置、被普通应用覆盖，并可展开完整面板。
- 任一文件失败不会阻断其他文件，界面不会把部分数据标记为完全实时。
- 应用继续只处理统计字段，不读取或保存对话正文。
