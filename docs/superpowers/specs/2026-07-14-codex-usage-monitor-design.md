# Codex Usage Monitor 设计说明

日期：2026-07-14  
状态：已确认，等待实现计划  
目标平台：macOS 14 及以上

## 1. 产品目标

构建一款原生 macOS 菜单栏应用，实时展示本机 Codex 使用情况：

- 5 小时限额剩余比例与重置时间；
- 周限额剩余比例与重置时间；
- 今日、近 7 天、全部累计 Token 消耗；
- 按本地项目归类的 Token 消耗；
- 总 Token 消耗；
- 低余量系统通知。

应用覆盖这台 Mac 上由 Codex 桌面 App、CLI 和 IDE 产生的本地会话。它只读取本机数据，不要求 OpenAI API Key，也不上传统计数据。

## 2. 已确认的产品决策

- 产品形态：原生 SwiftUI 菜单栏应用，使用 `MenuBarExtra`。
- 菜单栏信息密度：平衡型，常态显示 `5h 72% · 周 48%` 一类摘要。
- 展开面板：极简大面板，突出余量、重置时间、Token 总量和项目排行。
- 时间筛选：今日、近 7 天、全部累计，默认今日。
- 启动策略：首次启动询问是否随 Mac 登录启动；设置中可随时开关。
- 运行策略：选择随 Mac 登录常驻，不额外创建“监测 Codex 进程再启动应用”的后台助手。
- 通知策略：5 小时或周剩余量首次低于 20% 时提醒，低于 10% 时再提醒；同一限额窗口和阈值只提醒一次。
- 发布范围：首版供本机安装，使用临时签名；App Store、开发者证书签名和公证不在首版范围内。
- 产品与仓库名称：应用名为 `Codex Usage Monitor`，GitHub 仓库名为 `codex-usage-monitor-macos`。
- 最终交付：创建 GitHub 公开仓库，采用 MIT License，推送源码、测试、README 和构建说明。

## 3. 非目标

首版不包括：

- Codex 云端任务或其他设备的跨设备聚合；
- 团队、组织或多账号用量看板；
- OpenAI API 账单和 API Key 用量；
- Token 成本估算或货币换算；
- 对话正文、提示词、回复内容或项目文件内容分析；
- App Store 分发、Apple 公证或自动更新服务；
- Windows 或 Linux 客户端。

## 4. 用户体验

### 4.1 菜单栏

菜单栏常态显示两个核心余量，例如 `5h 72% · 周 48%`。状态颜色规则如下：

- 正常：系统前景色或蓝色状态；
- 剩余量低于 20%：对应值变为橙色；
- 剩余量低于 10%：对应值变为红色；
- 最新数据不可用或已过期：整体变灰。

点击菜单栏后打开大面板。

### 4.2 大面板

大面板从上到下包含：

1. 5 小时剩余比例、进度条和重置倒计时；
2. 周剩余比例、进度条和重置时间；
3. 今日 / 7 天 / 全部时间筛选；
4. 当前筛选范围的总 Token；
5. 按 Token 降序排列的项目列表；
6. 最新更新时间、设置入口和重建索引入口。

项目名默认取规范化工作目录的最后一级名称。若名称重复，则追加足以区分的父目录片段。长名称截断展示，完整路径通过辅助提示查看。

### 4.3 设置

设置页提供：

- 随 Mac 登录启动开关；
- 系统通知总开关；
- 20% 和 10% 两级通知开关；
- 数据源状态和最近一次成功更新时间；
- 重建本地统计索引；
- 打开使用说明和隐私说明。

## 5. 架构

应用按职责拆分为以下模块：

### 5.1 `CodexLogLocator`

定位 `$CODEX_HOME`。若未设置，则使用 `~/.codex`。发现活跃会话目录和归档会话目录，输出可监控的文件集合。

### 5.2 `SessionFileWatcher`

首次启动时扫描现有 JSONL 文件，之后通过文件系统事件监听目录变化。它为每个文件保存设备/文件标识、读取偏移和最后修改时间，只读取新增字节。若文件被截断、替换或移动，则重置该文件状态并安全重扫。

### 5.3 `CodexEventParser`

逐行解析 JSONL，只提取统计所需字段：

- 任务标识、创建时间和工作目录；
- `token_count` 事件时间；
- 单次或累计 input、cached input、output、reasoning output 和 total Token；
- `rate_limits` 的使用比例、窗口时长、重置时间和限额标识。

未知事件和未知字段被忽略。单行损坏不会阻止后续数据处理。

### 5.4 `UsageRepository`

通过系统 SQLite 保存可重建的本地索引。数据库至少包含：

- 会话与规范化项目路径；
- 按事件或时间桶保存的 Token 增量；
- 每个文件的增量读取游标；
- 最新有效限额快照；
- 每个限额窗口已经发送的通知阈值。

数据库位于应用的 Application Support 目录，不保存对话正文。

### 5.5 `UsageAggregator`

计算今日、近 7 天和全部累计数据，按规范化项目路径分组，并生成总量与项目排行。所有日期边界使用用户当前系统时区。

### 5.6 `UsageViewModel`

在主线程上向 SwiftUI 提供不可变视图状态，包括正常、无数据、加载中、数据过期和读取错误状态。

### 5.7 `NotificationCoordinator`

检测 20% 和 10% 阈值跨越，发送 macOS 系统通知。通知去重键由限额标识、重置时间和阈值组成；重置时间变化后开始新的通知窗口。

### 5.8 `LaunchAtLoginController`

使用 macOS `ServiceManagement` 管理登录启动，不安装额外的常驻进程监测助手。

## 6. 数据流

1. 应用启动并定位 Codex 本地数据目录。
2. 若数据库不存在或模式版本变化，扫描历史会话并建立索引。
3. 解析器从会话元数据获取任务、时间和工作目录。
4. 对每个 Token 事件计算新增 Token，并以事件时间写入本地索引。
5. 从最新有效事件更新 5 小时和周限额快照。
6. 文件监听器收到新增内容后，仅处理新增行。
7. 聚合器重新计算受影响的时间桶和项目，并发布新的视图状态。
8. 通知协调器检查是否跨越余量阈值。
9. SwiftUI 菜单栏和面板刷新；历史索引完成后，新数据应在 2 秒内显示。

## 7. 统计口径

### 7.1 Token 去重

Codex 日志可能包含同一任务的累计 Token 快照。实现不得直接相加所有累计快照。优先使用事件中的单次 Token 用量；若只有累计快照，则以同一任务上一次已处理累计值计算非负差值。

总 Token 以日志中的 `total_tokens` 为权威值。`input_tokens`、`cached_input_tokens`、`output_tokens` 和 `reasoning_output_tokens` 只作为明细，不重新相加推导总量，以避免不同日志版本中缓存或推理 Token 被重复计算。

若任务累计值下降，视为任务重置或日志轮换：记录新的基线，不产生负 Token，也不重复计入旧值。

### 7.2 时间范围

- 今日：系统当前时区当天 00:00 至当前时间；
- 近 7 天：当前时间向前连续 7 × 24 小时；
- 全部：本机可发现的全部活跃及归档会话。

### 7.3 项目归类

以会话记录的工作目录为主键。路径经过标准化，处理 `~`、符号链接和冗余路径段。没有工作目录的会话归入“未知项目”。

### 7.4 限额识别

从最新 `rate_limits` 快照读取使用比例和重置时间。以 `window_minutes` 识别 300 分钟的 5 小时窗口和 10080 分钟的周窗口；若服务端返回其他窗口，则保留服务端标签并避免错误标注为 5 小时或一周。剩余比例为 `max(0, 100 - used_percent)`。

限额是账户级快照，不按项目拆分；项目列表只拆分 Token 消耗。

## 8. 错误处理与恢复

- 数据目录不存在：展示首次使用空状态，并定期重试发现目录。
- 没有会话：展示“尚无 Codex 使用记录”，不显示错误。
- 单行 JSON 损坏：跳过该行并记录本地诊断信息。
- 日志文件截断或替换：重置该文件游标并重新解析，依靠稳定事件标识或增量规则避免重复。
- Codex 字段变化：忽略未知字段；关键字段缺失时保留最后一次可信统计并展示“数据已暂停”。
- 数据库损坏：保留损坏副本用于诊断，创建新库并从原始日志重建。
- 文件监听中断：退避重试，同时允许用户手动“立即重试”。
- 数据过期：显示最后成功更新时间并将菜单栏状态置灰，不把旧数据冒充实时数据。

## 9. 隐私与安全

- 应用不联网；GitHub 仓库中的程序不包含遥测服务。
- 不读取、保存或展示提示词、模型回复、工具输出和项目文件内容。
- 不读取或保存 OpenAI 登录凭据、Cookie、API Key 或访问令牌。
- 诊断日志不得包含对话正文；路径在面向用户的导出诊断中默认脱敏。
- SQLite 索引完全可由原始本地会话日志重建。

## 10. 测试与验收

### 10.1 单元测试

- 正常事件、未知事件、缺失字段、损坏 JSON 和扩展字段解析；
- 累计快照差值、任务重置和重复事件去重；
- 今日、7 天和全部时间边界；
- 路径规范化、同名项目区分和未知项目；
- 5 小时/周窗口识别、剩余比例和重置时间；
- 20%/10% 阈值跨越及窗口内通知去重。

测试夹具必须使用合成日志，不得提交真实用户会话。

### 10.2 集成测试

- 在临时目录创建模拟 `$CODEX_HOME`；
- 验证历史扫描结果；
- 向日志追加事件并验证 2 秒内更新；
- 验证文件截断、替换、移动和归档；
- 验证首次扫描与增量处理的最终结果一致；
- 验证索引删除后可以完全重建。

### 10.3 界面测试

- 正常、加载、无数据、数据过期和读取错误状态；
- 深色与浅色模式；
- 长项目名、大 Token 数和大量项目；
- 今日、7 天、全部筛选切换；
- 菜单栏正常、橙色、红色和灰色状态。

### 10.4 验收标准

- 合成夹具的项目统计和总量与人工计算完全一致；
- 首次索引完成后，新日志事件在 2 秒内出现在界面；
- 空闲状态不持续占用显著 CPU；
- 应用离线运行，不需要 API Key；
- 通知在同一窗口和阈值内只出现一次；
- 删除索引后可从日志恢复一致结果；
- 本机 `.app` 可以启动、常驻菜单栏并完成登录启动设置。

## 12. 风险与缓解

- Codex 本地日志格式不是本应用控制的稳定接口：解析器采用宽松字段读取、合成契约测试和最后可信值回退。
- 历史会话数量可能较大：首次扫描分批进行，后续使用文件游标和 SQLite 增量索引。
- 不同 Codex 客户端可能写入不同字段：以任务标识、工作目录、Token 与限额字段的能力检测为准，不假设所有事件完全一致。
- 限额规则可能改变：按窗口时长和服务端标签识别，不在代码中假设永远只有两种窗口。
- 公开仓库可能意外泄露本地数据：提交前运行敏感信息与仓库内容检查，只提交合成夹具。
